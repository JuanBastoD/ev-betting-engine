# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`ev-betting-engine`: a pre-match +EV (positive expected value) detection engine for football betting. It ingests "sharp" reference odds (Pinnacle via The Odds API) and team/player statistics (Sportmonks), compares local bookmaker odds (Playwright scraping) against fair-probability models (market devig, match xG, player-prop Poisson) to surface positive-EV bets, and exposes the whole thing through a FastAPI service with a periodic pipeline. Built strictly as Clean Architecture / DDD, one numbered "Prompt" (phase) at a time, each phase fully tested and committed before the next begins.

**Current state**: all four layers are wired end to end. Domain has all three quantitative cores (market-model devig/EV/Kelly, match-model Dixon-Coles xG, player-props Poisson) plus persistence and both API/scraping ingestion adapters (Phases 1-8). Phase 9 added `src/application/use_cases/` (Interactor per responsibility, ports-only dependencies) and `src/presentation/api/` (FastAPI + APScheduler + Composition Root) connecting everything to real infrastructure. Phase 10 (Level 1) added `src/domain/services/calibration/` - backtesting `SettledBet`s against Brier score/log loss/CLV/a calibration curve, and deriving versioned `CorrectionFactor`s applied *before* `calculate_ev` - plus the matching persistence/application/presentation wiring (`POST /value-bets/settle`, `GET /calibration/report`, `POST /calibration/factors/recompute`). `DixonColesModel`/`PoissonPropsModel` are still the concrete statistical models (Level 2 - a trained model - is scaffolded via `TrainableMatchModel`/`TrainablePropsModel` + `MLMatchModel`/`MLPropsModel` stubs, but deliberately not implemented yet; see README's "Gate de Nivel 2").

## Architecture

Four layers under `src/`, dependencies point strictly inward: `presentation -> application -> domain <- infrastructure`. **`src/domain/` never imports from any other layer** - this is checked manually at the end of every phase (walk every module, grep domain for infra/application/presentation imports) rather than enforced by tooling.

### Domain (`src/domain/`)

- `entities/`: frozen, slotted dataclasses (`@dataclass(frozen=True, slots=True)`), validated in `__post_init__`, raising `ValueError`. Entity identity is by id where one naturally exists (`Match.id`, `Team.id`); some entities nest full related entities rather than bare ids (e.g. `PlayerMatchStats.match: Match`, not `match_id: str`) so a provider adapter can build one from a single API response without a second lookup.
- `value_objects/`: same frozen/slotted/validated pattern for primitives (`DecimalOdds` > 1.0, `Probability` in [0,1], `EdgePercentage` >= -100, `Stake` > 0).
- `ports/`: ABCs only, no implementation. Two flavors: repositories (`MatchRepository`, `OddsRepository`, `ValueBetRepository`, `PlayerRepository`, `PlayerStatsRepository`) and external-data providers (`SharpOddsProvider`, `LocalOddsProvider`, `StatsProvider`, `PlayerStatsProvider`). `LocalOddsProvider` covers both the main match markets (`get_odds`) and player props (`get_player_props`).
- `PlayerPropMarket` deliberately carries a full `match: Match` reference (the lesson from the `OddsQuote` gap below) but a `player_name: str` instead of a `Player` entity: bookmakers expose only a display label, and resolving it to a known `Player` is application-layer matching, not ingestion. There is no repository port / ORM model / migration for `PlayerPropMarket` yet - persisting props needs its own decision when a use case wants to store them.
- `MarketType` gained `PLAYER_PROP` in Phase 8 (plain string column, no DB CHECK constraint - purely additive, no migration needed). `Selection` has no player reference of its own, so a player-prop `ValueBet.selection.outcome` compresses player name + prop type + Over/Under into one descriptive string (e.g. `"Lionel Messi SHOTS_ON_TARGET Over"`) rather than extending `Selection`/`ValueBet` - see `player_props/player_prop_detector.py`'s `_selection_for`. Flagged, not fixed, same posture as the gaps below: a first-class player reference on `Selection` would be the clean fix.

`OddsQuote` carries a full `match: Match` reference (fixed at the start of Phase 6 - it originally didn't, and `SqlAlchemyOddsRepository.save()` used to work around that with an optional `match_id` kwarg; that workaround is gone, `save()` now matches `OddsRepository`'s declared signature exactly and upserts the quote's own match). `ValueBet` gained a required `model_source: ModelSource` (`MARKET` / `STATISTICAL` / `BOTH` / `PLAYER_PROPS`) - which probability model (or combination) produced it - ahead of `MarketValueDetector`. `MATCH_STATS` was renamed to `STATISTICAL` and `BOTH` was added in Phase 7, before either had ever been produced by any code - a safe rename, not a breaking one.

`ValueBet` gained an optional `bookmaker: Bookmaker | None = None` field in Phase 10 - the second of the "flag, don't fix" gaps above to get resolved (after `lineup_confirmed` in Phase 9), once calibration needed to segment settled-bet accuracy per local bookmaker. All three detectors (`MarketValueDetector`, `MatchValueDetector`, `PlayerPropDetector`) now pass their originating `OddsQuote`/`PlayerPropMarket`'s `bookmaker` through. The remaining gap (`Selection` has no player reference of its own) is still open.

`src/domain/entities/bet_result.py`: `BetResult` (`WON`/`LOST`/`PUSH`) - the real-world outcome of a settled bet. `src/domain/entities/settled_bet.py`: `SettledBet` nests a full `ValueBet` (same "nest, don't reference by id" convention as `PlayerMatchStats.match`) plus `result`, `settled_at`, and an optional `closing_sharp_odds` (the sharp/Pinnacle price just before kickoff, captured for CLV). `SettledBet.profit_loss`/`SettledBet.clv` are computed properties, not stored columns - `profit_loss` is in the same unit as `ValueBet.suggested_stake.amount` (a bankroll fraction); `clv` is `closing_sharp_odds.implied_probability - local_odds.implied_probability`, `None` when no closing line was captured. There is no surrogate id on `ValueBet`, so `SettleBetUseCase` (Phase 10, application layer) looks one up by natural key (match id + selection + local odds) via the existing `ValueBetRepository.list_by_match_id` rather than requiring a new port method.

### Domain services (`src/domain/services/market_model/`)

The quantitative core comparing sharp (Pinnacle) odds against local bookmaker odds for match markets (1X2/Over-Under/BTTS). Pure domain service: no I/O, fully deterministic, no infrastructure imports (not even `scipy`/`numpy` - the two non-closed-form devig methods below use a 100-fixed-iteration bisection helper in `_bisection.py` instead).

- `devig.py`: Strategy pattern, one `DevigStrategy.devig(odds) -> list[Probability]` per overround-removal method - `MultiplicativeDevig`, `AdditiveDevig`, `ShinDevig`, `PowerDevig`. All four agree exactly in two hand-verifiable cases used throughout the tests: **no-vig** (raw implied probabilities already sum to 1.0 - every method is a no-op) and **symmetric-with-vig** (identical odds on every outcome - the fair split must be 1/n by symmetry, regardless of method). `MultiplicativeDevig`/`ShinDevig`/`PowerDevig` are mathematically guaranteed to always produce valid probabilities; `AdditiveDevig` is not - subtracting an equal *absolute* share of the overround can push a longshot's raw probability negative in a heavily skewed market (a known, documented weakness of the method, not a bug), which surfaces as a `ValueError` from `Probability`'s own invariant. This exact failure mode was found by the hypothesis property test before being pinned as a concrete example in `test_devig.py`.
- `ev_calculator.py`: `calculate_ev` is deliberately independent of `devig.py` - it only needs a `Probability`, from wherever it came, so the future team-form/player-prop statistical engines can call it directly. `exceeds_ev_threshold` takes `min_ev_threshold` as a plain float parameter (matching `Settings.min_ev_threshold`) rather than reading `Settings` itself, keeping the domain import-free.
- `kelly.py`: `kelly_stake` is generic (probability/odds/fraction/cap only) so it's reusable beyond the market model. `Stake.amount` here is a **fraction of bankroll** (e.g. 0.025 = 2.5%), not a currency amount - converting to a concrete stake size needs the user's actual bankroll, an application-layer concern for a later phase. `Stake`'s own invariant forbids `amount <= 0`, so "no bet" (Kelly's f* <= 0, or `kelly_fraction`/`max_fraction` configured to 0) returns `None` rather than `Stake(0.0)`.
- `detector.py`: `MarketValueDetector.detect(sharp_quotes, local_quotes)` orchestrates devig -> EV -> Kelly -> `ValueBet(model_source=ModelSource.MARKET)`. Takes a `DevigStrategy` instance by constructor injection. `sharp_quotes` must be every outcome of one match+market from one sharp bookmaker; every `local_quote` must belong to that same match+market and to an outcome the sharp side actually quoted, or `detect()` raises - mixing markets/matches is treated as a caller bug, not something to silently drop or skip. `fair_probabilities(sharp_quotes)` (devig-and-map, validated) and `require_same_market(...)` are public **so `MatchValueDetector` (Phase 7) can reuse them instead of re-deriving devig-orchestration logic** - `detect()` itself is just those two building blocks plus the EV/Kelly loop.

### Domain services (`src/domain/services/match_model/`)

The match-statistics quantitative core: derives its **own** opinion on 1X2/Over-Under/BTTS from team-form goal data (Dixon-Coles xG), independent of Pinnacle. Pure domain service, same purity/determinism bar as `market_model`.

- `team_strength.py`: `calculate_team_strength(form, league_average_goals, recent_form=None, recent_form_weight=0.6)` turns a `TeamForm` into attack/defense ratios relative to the league average. `TeamForm` (Phase 3) is an aggregate over up to 10 matches with **no per-match dates or venue split** - it cannot itself support "weight recent matches more" or "split home vs away performance". Both requirements are pushed to this function's *boundary* instead of into `TeamForm`: home/away split is just calling this function once with a home-only `TeamForm` and once with an away-only one (the caller slices by venue, this module doesn't know how); recency weighting is an *optional* second, shorter-window `TeamForm` (`recent_form`) blended in via `recent_form_weight` - omitting it still works, using the single `form` window alone.
- `absence_adjustment.py`: `apply_absence_adjustment` reduces a team's attack strength for key (forward/midfielder by default) players who are INJURED/SUSPENDED (full, configurable `reduction_per_key_absence`) or DOUBTFUL (discounted by `doubtful_weight`, and sets `has_unconfirmed_absences=True` on the result - mirrors the Phase 4 `estimate_start_probability` fallback pattern). "Key" also requires the player's goals+assists across supplied `PlayerMatchStats` to clear `min_goal_involvements`. Only touches attack (a missing striker/creative midfielder reduces goal output); defense is left untouched, a deliberately narrower scope. Propagating `has_unconfirmed_absences` into a persisted `ValueBet` is an application-layer decision beyond this domain service - there's no field on `ValueBet` for it, same "flag, don't fix" posture as the `ValueBet`-has-no-bookmaker gap above.
- `xg_model.py`: Strategy pattern (`MatchStatisticalModel.predict_match_probabilities`), current implementation `DixonColesModel` - bivariate Poisson with the Dixon & Coles (1997) tau correction for low-scoring dependence (0-0/1-0/0-1/1-1). Builds the full scoreline grid (truncated at `max_goals`, default 10, renormalized to sum to 1.0 exactly) and derives 1X2/Over-Under (configurable lines, default 1.5/2.5/3.5)/BTTS from it. Two hypothesis-found correctness issues, both fixed and pinned as regression tests: (1) individual grid cells clamped to `>= 0.0` before summing - the tau correction's `1 + lambda*rho` term goes negative once `lambda*abs(rho)` exceeds 1, reachable by multiplying together individually-reasonable attack/defense/home_advantage/league_average_goals inputs even though the resulting lambda (~10+) is unrealistic for football; (2) aggregate probabilities (`home_win`/`btts_yes`/etc.) clamped to `[0.0, 1.0]` since summing ~100 grid cells can overshoot by a sub-epsilon float rounding amount.
- `match_value_detector.py`: `MatchValueDetector` reuses `MarketValueDetector.fair_probabilities`/`require_same_market` for the market side and `calculate_ev`/`exceeds_ev_threshold`/`kelly_stake` for pricing - it adds only the statistical prediction and the `ConfirmationMode` policy. `CONFIRMATION` (default): a `ValueBet` (`model_source=BOTH`) is only emitted when market EV *and* statistical EV both individually clear `min_ev_threshold` against the same local odds, priced off their `market_weight`-blended probability. `INDEPENDENT`: statistical EV alone decides (`model_source=STATISTICAL`), `sharp_quotes` isn't required at all. The blend is never rechecked against the threshold after combining - EV is linear in probability, so a weighted blend of two probabilities that individually clear the bar mathematically must clear it too.

### Domain services (`src/domain/services/player_props/`)

The player-prop quantitative core: a player's own Over/Under probability on a metric (goals/shots-on-target/assists/cards) from historical per-90 rates, independent of both `market_model`'s devig and `match_model`'s xG - it reuses only `kelly_stake`/`calculate_ev` from the former and one narrow field (`TeamStrength.defense`, only valid for the GOALS prop type since that's the metric it's calibrated on) from the latter. Its own tiny Poisson pmf/cdf helpers are re-derived rather than imported from `xg_model.py`, keeping the two engines independently evolvable.

- `player_model.py`: Strategy pattern (`PlayerPropsModel.predict_probability`), current implementation `PoissonPropsModel` (Negative Binomial is a natural second implementation this interface leaves room for, not built). lambda = EWMA-of-per-90-rate * (expected_minutes / minutes_baseline) * opponent_strength_factor. The EWMA (`_ewma_per_90_rate`, weights `alpha*(1-alpha)**i` renormalized over however many usable matches are supplied) is what gives recent matches more weight - a plain average was explicitly ruled out by the prompt. Matches with `minutes_played == 0` are excluded (no usable rate), not treated as a rate of zero. `expected_minutes_from_lineup` blends `full_match_minutes`/`bench_minutes` by `LineupConfirmation.start_probability` - a *confirmed* slot (Phase 4: `start_probability` is exactly 1.0/0.0 when `is_confirmed`) collapses this to one or the other exactly; an estimated probability interpolates smoothly. `confidence_penalty` (unconfirmed lineup and/or DOUBTFUL/INJURED/SUSPENDED status, both configurable, compounding) feeds `confidence_adjusted_probability`, which blends the model's probability toward the odds' own breakeven (1/odds) rather than discounting the EV number after the fact - algebraically equivalent to "reduce the EV by X%" but expressible as a plain convex combination (needs no clamping, and can be fed straight back into `calculate_ev`/`kelly_stake` unchanged, keeping `ValueBet.fair_probability`/`edge`/`suggested_stake` mutually consistent).
- `prop_ev_calculator.py`: re-exports `calculate_ev`/`breakeven_odds`/`exceeds_ev_threshold` from `market_model.ev_calculator` unchanged (imported, not reimplemented) plus one prop-specific convenience wrapper, `calculate_prop_ev`, that takes a `PlayerPropMarket` directly instead of requiring the caller to unpack `.odds`.
- `player_prop_detector.py`: `PlayerPropDetector.detect(...)` prices exactly one `PlayerPropMarket` at a time (unlike the market/match detectors, which price many local quotes against one shared fair-probability point - each prop needs its own historical stats/lineup/injury context, so there's little to batch). Raises for a lineless prop (`line is None`) - out of scope for this line-based Poisson model. Returns `PlayerPropDetection` (a `ValueBet` paired with `lineup_confirmed`/`confidence`), **not** a bare `ValueBet` - `ValueBet` has no field for "how reliable was this input," so that signal rides alongside it instead of being silently dropped or forcing a schema change, same "flag, don't silently extend" posture as the `ValueBet`-has-no-bookmaker gap above. `model_source=ModelSource.STATISTICAL`, `selection.market_type=MarketType.PLAYER_PROP` always (`ValueBet` is intentionally the *same* entity type every other detector produces, not a fragmented parallel one).

Phase 10 (Level 2 scaffolding, not implemented): `xg_model.py` gained `TrainableMatchModel(MatchStatisticalModel)` (adds abstract `fit(training_data: Sequence[SettledBet])`/`model_version`) and a placeholder `MLMatchModel(TrainableMatchModel)` whose every method raises `NotImplementedError`; `player_model.py` gained the identical pair, `TrainablePropsModel(PlayerPropsModel)`/`MLPropsModel`. Both stubs exist purely so `MatchValueDetector`/`PlayerPropDetector` (which already depend only on the `MatchStatisticalModel`/`PlayerPropsModel` interface) need zero changes once a real trained model is built - see the README's "Gate de Nivel 2" for the explicit criteria (stable calibration over a full data cycle + ~300+ settled bets per segment) before that happens.

### Domain services (`src/domain/services/calibration/`)

Phase 10 (Level 1): backtests the three detectors' `ValueBet`s against real-world outcomes, independent of and reusable by all three (`market_model`, `match_model`, `player_props`) - pure domain service, same purity bar as the other three.

- `calibration_service.py`: `CalibrationService.calculate(settled_bets)` computes Brier score and log loss over non-`PUSH` settled bets only (a push has no won/lost signal to score a probability against - excluded from both, but *not* from `average_clv`, which is a market-timing signal independent of the bet's own outcome), a 10-bucket-by-default calibration curve (`_calibration_curve`, predicted-probability buckets vs. observed win frequency, `None`/`0` for empty buckets rather than a divide-by-zero), and `average_clv` over whichever settled bets captured a `closing_sharp_odds` (`None` if none did). `calculate_segmented(settled_bets)` additionally groups by `market_type`, bookmaker (skipping bets with no `ValueBet.bookmaker`), `model_source`, and (for `PLAYER_PROP` bets) prop type via the module-level `extract_prop_type(outcome)` helper - which re-parses `Selection.outcome`'s `"{player} {PROP_TYPE} {Over/Under}"` compressed string (the same one `player_prop_detector.py`'s `_selection_for` builds) rather than requiring a schema change.
- `correction_factor.py`: `CorrectionFactorService.compute_factors(settled_bets, computed_at)` computes one `CorrectionFactor` per segment (`"overall"` plus the same four segment types as above) as `observed_frequency / predicted_mean` over that segment's non-`PUSH` bets - segments under `min_sample_size` (constructor arg, `Settings.calibration_min_sample_size`, default 30) or with a `predicted_mean` of exactly 0.0 (nothing to divide by) are skipped, not fabricated. `CorrectionFactor` is a frozen dataclass with its own `computed_at`/`data_range_start`/`data_range_end` (the min/max `settled_at` among the segment's own bets) - every `compute_factors()` call is meant to persist a **new** row (see `CorrectionFactorRepository`/`SqlAlchemyCorrectionFactorRepository` below), never overwrite a previous one, so the correction history per segment stays auditable. `apply_correction_factor(probability, factor)` is a pure multiply-and-clamp-to-[0,1] function - the explicit adjustment step meant to run **before** `calculate_ev`, never inside `DixonColesModel`/`PoissonPropsModel` themselves. `latest_by_segment(factors)` reduces a list down to only the most-recently-`computed_at` factor per `(segment_type, segment_value)` pair, for a caller that wants "the current correction," not the whole version history.

### Infrastructure - persistence (`src/infrastructure/persistence/`)

- `models.py`: every SQLAlchemy `DeclarativeBase` model in one file, deliberately kept separate from domain dataclasses. Includes `UTCDateTime`, a `TypeDecorator` that normalizes to UTC on both read and write - required because SQLite/aiosqlite silently drops tzinfo, which would otherwise violate the domain's tz-aware invariant on timestamps.
- `mappers.py`: explicit, pure `entity_to_model` / `entity_from_model` functions - the only file allowed to import both `src.domain` and `models.py`.
- `upserts.py`: shared get-or-create/update helpers (`upsert_team`, `upsert_league`, `upsert_bookmaker`, `upsert_match`, `upsert_player`) for reference entities that have no dedicated repository port but are nested inside ones that do (e.g. saving a `Match` transitively upserts its two `Team`s and `League`).
- `repositories/`: one file per port, e.g. `SqlAlchemyMatchRepository(MatchRepository)`. **Repositories only `flush()`, never `commit()`/`rollback()`** - the session is the unit of work, and whoever owns the session (a future use case) owns the transaction boundary.
- `session.py`: `get_session()` is the injectable async-generator dependency; the engine/sessionmaker are lazily created once from `Settings()` and cached at module level (`reset_session_factory()` for tests/shutdown).
- Alembic (`alembic/`) is async-mode, `env.py` imports `Base.metadata` as `target_metadata` and reads the DB URL through `src.infrastructure.config.Settings` (env vars/`.env`) instead of duplicating it in `alembic.ini` - switching SQLite to Postgres is only ever a `DATABASE_URL` change. `env.py` also has a `render_item` hook so the custom `UTCDateTime` type renders as plain `sa.DateTime(timezone=True)` in generated migrations (otherwise autogenerate emits an unimported reference to the application class).
- Phase 10 additions (migration `d9433886bf4e`): `value_bets.bookmaker_id` (nullable FK to `bookmakers`) plus two new tables. `SettledBetModel` is **deliberately denormalized** - a full copy of the settled `ValueBet`'s describing columns (not a `value_bets.id` FK) so a settled bet stays self-contained even if the originating `value_bets` row is later pruned; `SqlAlchemyValueBetRepository`/`SqlAlchemySettledBetRepository` both upsert `Bookmaker` via the existing `upsert_bookmaker` only when `ValueBet.bookmaker is not None`. `CalibrationFactorModel` mirrors `CorrectionFactor` exactly (one row per version, `id` is a plain surrogate key, never reused as an update target). Because SQLite can't `ALTER TABLE ADD COLUMN ... FOREIGN KEY` or drop constraints directly, the `value_bets.bookmaker_id` column/FK addition uses `op.batch_alter_table(...)` (same convention as the earlier `lineup_confirmed` migration, `d4e1f9a2b613`) - the two new tables are plain `create_table` since SQLite has no trouble creating a table with a FK from the start.

### Infrastructure - external providers (`src/infrastructure/providers/`)

Two independent provider integrations, each following the same **Adapter + DTO + Mapper** shape and both under `src/infrastructure/providers/api/`:

1. **`api/`** (top-level files) - The Odds API (`TheOddsApiClient`, sharp/team-level odds and scores). Auth via `apiKey` query param.
2. **`api/player_stats/`** - Sportmonks (`SportmonksClient`, player-level stats/injuries/lineups). Auth via `api_token` query param; every response is wrapped in a `data` envelope that must be unwrapped before DTO validation.

Each integration has this internal shape - look at the sibling module before adding a third provider:
- `dtos.py`: pydantic models mirroring the raw JSON, `extra="ignore"` everywhere. Nothing outside the provider's own directory is allowed to see these.
- `client.py`: wraps `httpx.AsyncClient` (configurable timeout/pooling, async context manager). Retries via `tenacity.AsyncRetrying` **scoped to HTTP 429/5xx only** - other httpx errors, non-429 4xx, and pydantic `ValidationError`s on the response all become `ProviderUnavailableError` immediately (no retry); 429 becomes `RateLimitError` (carrying `Retry-After` when present) once retries are exhausted. Both exceptions live in `providers/exceptions.py`, shared across every provider so no `httpx`/`pydantic` exception ever leaks past this layer. Wait/attempt parameters are constructor args (not baked into a decorator) specifically so tests can pass near-zero backoff instead of sleeping for real.
- `mappers.py`: pure `dto_to_entity` functions, no I/O.
- The concrete provider class (e.g. `TheOddsApiSharpOddsProvider`, `SportmonksPlayerStatsProvider`) implements the matching domain port and is the only place that calls both `client.py` and `mappers.py`.

The two clients are standalone (no shared base class) by deliberate choice, to avoid touching the already-shipped/tested odds client when Sportmonks was added - revisit if a third provider makes the duplication worth factoring out.

Every provider adapter carries enough context in its own DTOs to build a full `Match` (home/away team names+ids, league, kickoff) from a single response, rather than requiring a separate lookup - this was a deliberate correction after the `OddsQuote` gap above was noticed.

`httpx.AsyncClient(base_url=...)` + a request path starting with `/` silently drops any path segment in `base_url` (e.g. the `/v4` in `https://api.the-odds-api.com/v4`), per RFC 3986 URL-join rules. Both clients normalize `base_url` to end with `/` and build request paths *without* a leading `/` to avoid this - keep that convention if you touch either client.

### Infrastructure - scraping (`src/infrastructure/providers/scraping/`)

Playwright-based adapter for local Colombian bookmakers (Betplay, Stake, Betano), implementing `LocalOddsProvider`. Patterns: Page Object Model + Factory + Template Method.

- `base.py`: `AbstractBookmakerScraper` owns the whole shared flow (rate-limit delay -> `goto` + `wait_for_selector` with tenacity retries -> `inner_html` -> parse). Playwright touches exactly three page operations (`goto`, `wait_for_selector`, `inner_html`/`click`) so tests fake the Page trivially. Retry/backoff/delay/timeouts are constructor args (same reasoning as the API clients). 1X2 parsing is position-based (first=Home, second=Draw, third=Away), not name-based - sites abbreviate team names unpredictably.
- One module per bookmaker (`betplay.py`, `stake.py`, `betano.py`): all of a site's CSS selectors, label vocabulary (Spanish/English, over/under prefixes, prop-type names) and URL scheme live in its class only. Each registers itself with `@ScraperFactory.register`; `scraping/__init__.py` imports all scrapers so importing the package wires the registry - a new bookmaker is one new subclass + one import line, no orchestrator changes.
- **Parsing is 100% pure**: scrapers grab container `inner_html` and hand the string to `parse_match_odds`/`parse_player_props`, which run on `html_utils.py` - a tiny stdlib-only (html.parser) fragment parser written to avoid adding beautifulsoup/selectolax as a dependency. This is what lets the whole parse layer be tested against local `.html` fixtures with `page=None`.
- Unrecognized market/prop labels are *skipped*, not errors (sites offer far more markets than the domain models); missing structural blocks raise `SelectorNotFoundError`, malformed odds text raises `OddsParsingError`. All scraping exceptions extend `ScrapingError`, which extends the shared `ProviderError` - no Playwright exception may leak past this package.
- `browser.py`: `PlaywrightBrowserSession` is the only place that launches Chromium (headless, realistic user-agent/viewport); its `close()` releases context/browser/driver even if a step fails. `provider.py`: `PlaywrightLocalOddsProvider` is scoped to one bookmaker per instance (mirroring the sharp provider's one-sport_key scoping), opens a fresh Page per call and always closes it.
- Real scraping needs `uv run playwright install chromium` once; **tests never launch a browser** and don't need it. README carries the ToS/Coljuegos legal note - keep delays conservative and configurable.

### Application (`src/application/`)

- `use_cases/`: one Interactor per responsibility, constructor-injected, depending only on domain ports (`SharpOddsProvider`, `MatchRepository`, etc.) - never a concrete `TheOddsApiClient`/`SqlAlchemyMatchRepository`/etc. Each `execute()` takes **one** `Match` (not a list) - the pipeline facade loops over matches calling it once per match, which is also exactly what `/value-bets/query`'s single-match on-demand flow needs, so both paths share the same per-match use cases with zero duplication.
  - `IngestSharpOddsUseCase`/`IngestLocalOddsUseCase`/`IngestPlayerStatsUseCase`: persist what already has a repository port (`Match`, `OddsQuote` main-market quotes, `Player`, `PlayerMatchStats`) and *return* what doesn't (`TeamForm`, `PlayerPropMarket`, `InjuryStatus`, `LineupConfirmation` - all "fetched fresh each run, not archived," the same pattern established since Phase 5/7) for the next step to consume directly, no repository re-fetch needed in between.
  - `DetectMatchValueBetsUseCase`: groups `sharp_quotes`/`local_quotes` by `(market_type, line)` and calls `MatchValueDetector.detect()` once per market group (the domain detector, like `MarketValueDetector` before it, prices exactly one market per call) - added `MatchValueDetector.mode` as a new public property specifically so this use case can skip a market with no sharp coverage in CONFIRMATION mode (nothing to confirm against) while still pricing it in INDEPENDENT mode (sharp coverage was never required there).
  - `DetectPlayerPropValueBetsUseCase`: resolves each `PlayerPropMarket.player_name` (a bookmaker display label, no `Player` reference - flagged since Phase 5) to a known `Player` by case-insensitive exact name match against whoever appears in that match's lineup/injury data - this **is** the "later application-layer matching step" Phase 5's docstring anticipated. A prop for a name we have no data on is skipped, not an error.
  - `RunPipelineUseCase` (**Facade**): the five use cases above, chained. `execute(matches=None)` defaults to `MatchRepository.list_upcoming()` (the scheduled/periodic run); an explicit list (even one match) is the exact same code path the on-demand query endpoint uses.
  - `ListValueBetsUseCase`: `ValueBetRepository`'s port only exposes `list_all()`/`list_by_match_id()`, no server-side filtering - filters (league/EV/date/market_type/model_source) are applied in plain Python over the full result set. Doesn't scale to a huge table; an explicit, documented tradeoff rather than extending the port for this phase.
  - `SettleBetUseCase` (Phase 10): looks a `ValueBet` up by natural key (match id + selection + local odds) via `ValueBetRepository.list_by_match_id`, pairs it with a real-world `BetResult` into a `SettledBet`, and persists it via `SettledBetRepository`. Raises `ValueBetNotFoundError` (404) when no `ValueBet` matches exactly - a caller settling the wrong odds/outcome is treated the same as "doesn't exist," not silently ignored.
  - `GetCalibrationReportUseCase` (Phase 10): fetches every `SettledBet`, applies the same "filter in plain Python" tradeoff as `ListValueBetsUseCase` (optional `model_source`/`market_type` pre-filter, no port change), then hands the result to `CalibrationService.calculate_segmented`.
  - `ComputeCorrectionFactorsUseCase` (Phase 10): fetches every `SettledBet`, calls `CorrectionFactorService.compute_factors`, and persists each resulting `CorrectionFactor` via `CorrectionFactorRepository` - one new versioned batch per call, per the repository's own "never update in place" contract.
  - `src/application/exceptions.py`: `MatchNotFoundError`/`PlayerPropNotFoundError`/`ValueBetNotFoundError` (Phase 10) - orchestration-level "the thing you asked about doesn't exist" errors, distinct from `ProviderError` (upstream failure) and from the domain's own plain `ValueError` (invariant/business-rule violations).

Known, deliberately-flagged (not fixed) match-model limitation surfaced by wiring real data through it: `StatsProvider.get_team_form(team)` returns one combined `TeamForm` per team (no home/away split), even though `calculate_team_strength` was designed to accept separate home-only/away-only forms (Phase 7). `DetectMatchValueBetsUseCase` currently passes the same combined form for both the home and away calculation calls - correct given the data actually available, but leaves the venue-split capability unused. Extending `StatsProvider`/`TheOddsApiStatsProvider` to return split forms would be the clean fix.

### Presentation (`src/presentation/api/`)

FastAPI. Patterns: Composition Root, Dependency Injection, Facade (the pipeline, in `application/`).

- `dependencies.py` is the **Composition Root** - the only module that imports concrete infrastructure (`SqlAlchemyMatchRepository` et al., `TheOddsApiClient`, `SportmonksClient`, `PlaywrightLocalOddsProvider`) or concrete statistical models (`DixonColesModel`, `PoissonPropsModel`). Two layers, deliberately separate (see the module's own docstring for the full rationale): `build_*` functions take plain arguments (`session`, `Settings`, a provider) and do the actual wiring, knowing nothing about FastAPI; `get_*` functions are thin `Depends()` adapters over them. This split exists because `scheduler.py` needs the *exact same* use-case graph *outside* a FastAPI request - `Depends(...)` default values only resolve through FastAPI's own DI machinery, not through a plain function call, so the scheduler calls `build_run_pipeline_use_case` directly with its own session/provider lifecycle instead. **Prompt 10 swap points**: `build_match_value_detector`/`build_player_prop_detector` construct `DixonColesModel()`/`PoissonPropsModel()` on one line each, explicitly commented - a trained model just replaces that one line, since everything downstream depends on the `MatchStatisticalModel`/`PlayerPropsModel` interface already. Phase 10 added `build_calibration_service`/`build_correction_factor_service` (read `Settings.calibration_bucket_width`/`calibration_min_sample_size`) and matching `get_settle_bet_use_case`/`get_calibration_report_use_case`/`get_compute_correction_factors_use_case` adapters, following the exact same two-layer split.
- `app.py`: `create_app()` builds the `FastAPI` instance (routers + exception handlers); the module-level `app` is what `uv run uvicorn src.presentation.api.app:app` serves. `lifespan()` eagerly initializes the DB connection pool (`get_session_factory()`) and starts the `AsyncIOScheduler` on startup, tearing both down on shutdown - it does **not** create tables (that's `alembic upgrade head`'s job, run separately).
- `scheduler.py`: `create_scheduler(settings)` registers one interval job (`Settings.pipeline_interval_seconds`) that calls `run_scheduled_pipeline`, which opens its own DB session and `PlaywrightBrowserSession` for that one pipeline pass only (nothing held open between ticks) and calls `build_run_pipeline_use_case` - the same composition-root function `get_run_pipeline_use_case` (the API's `Depends()` adapter) calls, so the periodic job and the manual `/pipeline/run` endpoint are guaranteed to behave identically.
- `exception_handlers.py`: registered once on the app, not per-route. `MatchNotFoundError`/`PlayerPropNotFoundError`/`ValueBetNotFoundError` (Phase 10) -> 404; plain `ValueError` (how every domain entity/service reports an invariant or business-rule violation, Phase 1 onward - there is no domain-specific exception hierarchy) -> 400; `ProviderError` (upstream odds/stats API or scrape failure) -> 502; anything else -> 500 with the real exception logged server-side but never leaked in the response body. Every branch logs via `structlog` before responding.
- `logging_config.py`: `configure_logging()` sets up `structlog` with a JSON renderer, called once in `lifespan()` - every module just does `structlog.get_logger(__name__)`, no per-module setup.
- `schemas.py`: the only place a domain `ValueBet` gets flattened into JSON - `ValueBetSchema.from_entity()`. `lineup_confirmed` is always present on the schema but only non-null for player-prop bets (see the `ValueBet.lineup_confirmed` note above); `bookmaker` (Phase 10) is the bookmaker's name, or `null`. Phase 10 also added `SettleBetRequest`/`SettleBetResponse` and the calibration-report schemas (`CalibrationBucketSchema`/`CalibrationMetricsSchema`/`CalibrationReportResponse`/`CorrectionFactorSchema`/`ComputeCorrectionFactorsResponse`), each with its own `from_entity()` mirroring the domain dataclass field-for-field.
- `routers/`: `health.py` (`GET /health`), `pipeline.py` (`POST /pipeline/run`, `POST /value-bets/query` - both call the identical `RunPipelineUseCase`, `query` additionally resolving `match_id` via `MatchRepository` first and post-filtering by `player_name`/`prop_type` against `Selection.outcome`'s composed string), `value_bets.py` (`GET /value-bets`, thin wrapper over `ListValueBetsUseCase`), `calibration.py` (Phase 10: `POST /value-bets/settle`, `GET /calibration/report`, `POST /calibration/factors/recompute` - thin wrappers over `SettleBetUseCase`/`GetCalibrationReportUseCase`/`ComputeCorrectionFactorsUseCase`).

`ValueBet` gained an optional `lineup_confirmed: bool | None = None` field in Phase 9 (nullable column, migration `d4e1f9a2b613`) - the first of the three "flag, don't fix" `ValueBet`/`Selection` gaps (Phases 6-8) to actually get resolved, once `GET /value-bets` gave a concrete, unavoidable need to recover this signal from *persisted* bets (the transient `PlayerPropDetection` wrapper that used to be the only place it lived doesn't survive a repository round-trip). `PlayerPropDetector` now sets it directly; every other `ValueBet` construction site is untouched (the field defaults to `None`). Phase 10 resolved the second gap (`ValueBet.bookmaker`, see the domain-entities section above); the last remaining one (`Selection` has no player reference) is still open.

### Testing conventions

- HTTP is always mocked with `respx` (`with respx.mock(assert_all_called=True) as router: ...`), never real network - respx's default `assert_all_mocked=True` makes an accidental real call fail loudly.
- Persistence tests use an in-memory SQLite engine with `poolclass=StaticPool` (required - without it, each connection checkout would see its own empty `:memory:` database) and a session bound to a connection-level transaction that's rolled back at teardown for isolation, even though repositories only flush.
- Retry-related client tests construct the client with near-zero `wait_min`/`wait_max`/`wait_multiplier` so exhausting retries doesn't add real seconds to the suite.
- Fixtures for external APIs live in `fixtures/*.json` next to their tests and are meant to be realistic sample payloads, not minimal stubs - reuse and extend them rather than inlining ad hoc dicts when adding coverage for the same endpoint.
- Scraping tests never launch a browser or touch the network: Playwright's `Page` and the `async_playwright()` stack are replaced by hand-rolled fakes in `tests/infrastructure/providers/scraping/fakes.py` (failure queues let a test script "fail twice, then succeed" to exercise retries without real waits). The `.html` fixtures for the three bookmakers encode the *same* logical odds content in each site's own markup/language/decimal format, so `test_scrapers_parsing.py` parametrizes one expected result across all scrapers - keep that equivalence when extending them.
- Hypothesis (`@given`) is used throughout both `services/market_model` and `services/match_model` for invariant tests (probabilities sum to 1, stay in [0,1], stakes never negative). It flags function-scoped pytest fixtures used under `@given` as a health-check failure (not reset between generated examples) - for entities that are immutable/side-effect-free to construct (a `Team`, a `Match`), build them as plain module-level constants instead of fixtures in those specific test functions, per `tests/domain/services/match_model/test_xg_model.py`/`test_absence_adjustment.py`.
- All three quantitative-core test suites hand-derive expected values independently of the implementation (exact fractions for devig, direct transcriptions of the Dixon-Coles formula for xG, closed-form Poisson CDF arithmetic for player props) rather than re-running the source under test - see the module docstrings in `test_devig.py`/`test_xg_model.py`/`test_player_model.py` for the derivations. Several hypothesis-found edge cases across the three modules were fixed and pinned as concrete regression tests rather than only living in the property test that found them.
- `tests/domain/services/calibration/` (Phase 10) follows the same hand-derivation convention: Brier score/log loss are computed by hand in `test_calibration_service.py`'s module docstring for a 3-bet example, CLV from a plain implied-probability difference, and the calibration curve against a synthetic dataset with a fully known expected bucket distribution (specific buckets non-empty, everything else asserted empty) rather than just checking aggregate counts. `MLPropsModel`/`MLMatchModel` tests (in `test_player_model.py`/`test_xg_model.py`) assert every method raises `NotImplementedError` - deliberately verifying the stub fails loudly rather than silently returning a bogus prediction if instantiated by mistake.
- `tests/fakes.py` (top-level, not per-package) holds one in-memory fake per domain port, shared by `tests/application/`'s use-case unit tests and `tests/presentation/api/`'s endpoint tests via `app.dependency_overrides` - the one deliberately cross-cutting test module, since both layers need the exact same fakes.
- `tests/presentation/api/conftest.py`'s `app` fixture overrides `get_run_pipeline_use_case`/`get_list_value_bets_use_case`/`get_match_repository` wholesale (not the individual provider-level dependencies they'd use in production) because `build_run_pipeline_use_case` constructs the real API/scraping providers *inside itself*, not via separate `Depends()` functions - see `dependencies.py`'s docstring. The override still uses **real** `SqlAlchemy*Repository`s against an in-memory SQLite session (same `StaticPool` pattern as the persistence tests) plus the real `build_match_value_detector`/`build_player_prop_detector` detector wiring - only the four external-data providers (sharp odds, stats, local odds, player stats) are faked.
- **`[tool.coverage.run]` needs `concurrency = ["thread", "greenlet"]`.** Without it, coverage silently drops every line that runs *after* an `await` on a SQLAlchemy async call made through a FastAPI request (reproduced down to a 15-line minimal repro: a plain route awaiting `session.execute()` loses all coverage on the lines following it, while the identical code path invoked directly in a non-HTTP unit test is measured correctly). Root cause: SQLAlchemy's async engine bridges to its sync core via `greenlet`, which coverage.py's default concurrency tracking (under Python 3.12's `sys.monitoring`-based tracer) doesn't follow unless told to. The code was never actually untested - only unmeasured - but don't remove this setting or the false gap comes back silently.
